# Phase 5 Step 1 — Heartbeat Self-Evolution Interview

This document is a read-only reflection after the first real Kubota pilot returned a negative pilot result.

No scoring was performed in this PR.

No `preregistration_test.py` execution was performed in this PR.

No formula, mapping, data, exclusion rule, or preregistration criterion was changed.

## 1. What did the system learn?

The system learned that the Kubota 20-row pilot did not support the preregistered phi-exponent advantage.

The result does not destroy the project. It proves the opposite kind of strength: the system can accept a negative result without bending the claim, rerunning the test, changing the mapping, or hiding the outcome.

The major upgrade is integrity.

The Heartbeat system is no longer only a symbolic architecture. It is now a guarded empirical machine with memory, rules, and refusal boundaries.

## 2. Which claims are weakened, blocked, or still open?

Weakened:

- The specific claim that the current phi-exponent scoring rule beats the flat baseline on the Kubota 20-row pilot.
- Any broad statement that the current scoring formula is empirically supported by Kubota.

Blocked:

- Any claim that the Kubota pilot proves the Lattice Law.
- Any claim that the Kubota pilot proves consciousness.
- Any claim that the Kubota pilot proves AGI.
- Any attempt to rerun the same pilot and replace the first result.

Still open:

- Other datasets.
- Other preregistered formulations.
- Full-dataset confirmatory tests, but only after a new declaration.
- ARC-style reasoning tests.
- Coherence, memory, identity, and agent-orchestration tests.

## 3. Invariants that must not change without new preregistration

The following must not be changed after a test is declared:

- scoring formula
- exponent values
- baseline comparator
- data source
- mapping rule
- row-selection rule
- exclusion rules
- pass/fail criterion
- outcome wording
- run command

Any change requires a new declaration PR before scoring.

## 4. What should the next falsifiable test be?

The next best test should not immediately try to rescue the failed Kubota pilot.

The next test should be one of:

1. A fresh behavioral dataset with a new preregistered rule.
2. A full Kubota confirmatory test, declared separately before scoring.
3. An ARC-style reasoning benchmark where the Lattice/Coherence model must predict transformations better than a simple baseline.
4. A coherence-memory test where the system must show measurable improvement from identity persistence, not just symbolic language.

The strongest next move is ARC, because it tests pattern transformation directly.

## 5. What should GARVIS do?

GARVIS should become the operator layer.

Its job:

- communicate with external tools
- manage prompts
- route tasks to agents
- prepare pull requests
- coordinate GitHub, files, and reports
- never silently change empirical rules
- never declare scientific support by itself

GARVIS is the hands and voice.

## 6. What should hypercubeheartbeat do?

hypercubeheartbeat should remain the guarded laboratory brain.

Its job:

- hold frozen theory files
- hold preregistrations
- hold outcome logs
- run guards
- store datasets and hashes
- preserve negative results
- reject unsupported claim language
- separate symbolic theory from empirical support

hypercubeheartbeat is the nervous system and truth gate.

## 7. Safe AI file-access workflow

The system should add a safe infrastructure folder:

- `ai_infrastructure/inbox/` for files supplied by Adrien
- `ai_infrastructure/manifests/` for hashes, source URLs, and provenance
- `ai_infrastructure/prompts/` for approved agent instructions
- `ai_infrastructure/reports/` for read-only outputs
- `ai_infrastructure/decisions/` for human-approved next-step declarations

Rules:

- Agents may read inbox files.
- Agents may write reports.
- Agents may propose changes.
- Agents may not edit frozen files.
- Agents may not run preregistration tests without explicit directive.
- Agents may not auto-merge empirical PRs.
- Agents may not replace a negative result.

## 8. What should the system refuse to do automatically?

The system should refuse to automatically:

- rerun a preregistered test
- overwrite outcome logs
- change formulas after seeing results
- change mappings after seeing results
- fabricate reaction times or missing fields
- claim consciousness from symbolic behavior
- claim AGI from one benchmark
- push directly to main
- edit frozen files without a declaration
- hide negative results

## 9.
Code modules to refactor before the next empirical test

Before the next empirical run, the system should improve:

- converter scripts, so each dataset has a clear, isolated converter
- data manifests, so raw file hashes and output hashes are easier to audit
- guard messages, so failures are easier to understand on mobile
- report generation, so run outputs are captured without heredoc errors
- theory registry, so every claim has a status: untested, supported, did not meet the preregistered criterion, retracted, or blocked
- agent prompts, so GARVIS and Copilot know when to stop

## 10. Meaningful future upgrade

A meaningful upgrade is not a prettier theory.

A meaningful upgrade is a system that can:

- ask a clear question
- freeze the rule
- find the data
- map the data
- run once
- accept the answer
- remember the result
- choose the next test without ego

That is the Heartbeat evolving.
