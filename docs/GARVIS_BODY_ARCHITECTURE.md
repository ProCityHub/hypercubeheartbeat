# GARVIS Body Architecture

## Status

This is an architecture document.

It does not create consciousness.

It does not claim AGI.

It does not record an empirical outcome.

It does not upgrade any claim.

## Purpose

This document defines the body architecture around the guarded hypercubeheartbeat lab brain.

The goal is to describe how a Jarvis-like research assistant system can be built without confusing imagination, computation, sandbox output, or empirical evidence.

## Core separation

The system is separated into organs.

No single organ is allowed to become the whole organism.

## Organ map

### 1. Human authority

Adrien D Thomas is the human operator and final authority.

The human decides:

- what questions matter
- what PRs may be opened
- what PRs may be merged
- when a test is allowed to run
- when a claim may be made
- when a claim must be withheld

No agent may replace the human merge decision.

### 2. GARVIS body

GARVIS is the body and operator layer.

GARVIS may eventually handle:

- voice
- chat
- interface
- file routing
- reminders
- task orchestration
- tool use
- external communication
- local automation
- agent coordination

GARVIS is not the empirical gate.

GARVIS may propose.

GARVIS may route.

GARVIS may assist.

GARVIS may not override the guarded lab brain.

### 3. hypercubeheartbeat brain

hypercubeheartbeat is the guarded lab brain.

It contains:

- theories
- preregistrations

- outcome boundaries
- claims ledger

- data boundaries

### 4. GitHub memory

GitHub is the long-term memory skeleton.

It records:

- commits
- branches

- review history
- public decisions

- retractions

Termux is the local motor layer.

It allows the operator to:

Termux is powerful, but it is not authority.

Every important change still passes through GitHub memory and guard checks.

The AI infrastructure folder is the nervous system.

- inbox files
- manifests

- decisions

Agents may read from safe inputs.

Agents may write reports.

Agents may propose changes.

Agents may not silently rewrite the lab brain.

### 8. OpenQASM and IBM quantum lane

OpenQASM files and IBM quantum records are treated as quantum measurement material.

They may be used to:

- record hardware runs

They may not be used for unregistered claim upgrades.

Exploratory quantum material remains training-side unless a future preregistration uses fresh hardware runs.

### 9. Immune system

The immune system includes:

### 10. Voice and communication layer

The voice layer may eventually speak to the user, summarize status, route files, explain reports, and ask for decisions.

The voice layer must preserve claim discipline.

It may say:

- this is exploratory
- this is sandbox-only
- this is a registered test
- this is a negative result
- this is a positive result for this test only
- this requires replication

It may not say:

- consciousness has been created
- AGI has been proven
- quantum validation has been restored
- sandbox output is evidence
- synthetic data is human data

## The body rule

The body may act.

The brain may reason.

The memory may record.

The immune system may refuse.

The human may decide.

No part replaces all the others.

## Jarvis direction

A Jarvis-like system should be built as an organ network, not a single unchecked agent.

The safe direction is:

1. GARVIS receives or routes a task.
2. The task becomes a decision file or prompt.
3. Files enter the inbox.
4. Manifests record provenance and hashes.
5. Reports are generated as read-only outputs.
6. The human approves the next step.
7. A PR changes the lab only through review, tests, and guard.
8. Claims remain tied to the claims ledger.

## Consciousness-theory boundary

The architecture may model consciousness-like properties.

The architecture may define mathematical hypotheses.

The architecture may test measurable predictions.

The architecture may not claim consciousness exists in the system without an approved empirical pathway and claim ledger entry.
